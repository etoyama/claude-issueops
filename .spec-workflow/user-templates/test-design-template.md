## Test Levels

| Level | Name | Automation | Tools |
|-------|------|------------|-------|
| L1 | [構造検証 / 単体テスト 等] | 自動 | pytest / vitest / ruff / mypy |
| L2 | [スクリプト単体 / 統合テスト 等] | 自動 | pytest / vitest |
| L3 | [Integration Verification / E2E 等] | 半自動・手動 | shell / Playwright MCP / 手動確認 |

> **Level の選び方**: 機能の性質による。Claude Code plugin / CLI tool なら「構造検証 / スクリプト単体 / Integration Verification」が適切。Backend / Web app なら「Unit / Integ / E2E」。Strategy Overview で選択理由を述べること。

## Key Testing Decisions

> 「なぜこのテスト戦略にしたか」を明示する。レビュアと未来の自分のため。スコープ外の判断も書く。

1. **[決定 1]**: [説明 — どんな選択をしてなぜそうしたか。例: 「hook スクリプトの既存テストは移植しない: 既存ロジック変更なし、構造検証 (L1) でスクリプト参照の整合性のみ担保」]
2. **[決定 2]**: [説明 — 例: 「inject-rules.py は独立テスト可能に設計: stdout に JSON を出力するだけのスクリプトなので、subprocess で呼び出してテスト可能」]
3. **[決定 3]**: [説明 — 例: 「E2E は手動チェックリスト: マーケットプレイス経由インストールはネットワーク依存、CI で安定実行不可」]
